# 后端持久化闭环回归测试

日期：2026-05-31

## 1. 测试目标

本轮只验证“新增记账数据能从前端写入后端 SQLite，并在刷新、重启后保持一致”的闭环。

重点范围：

- SQLite 数据库自动创建、建表、默认种子数据初始化。
- `GET /api/accounts` 返回真实账户数据。
- `GET /api/categories` 返回真实收入/支出分类。
- `GET /api/transactions` 返回真实账单时间轴。
- `POST /api/transactions` 可新增收入、支出。
- `POST /api/transfers` 或 `POST /api/transactions` 的转账入口可生成关联转账数据。
- 前端首页、记一笔、资产、报表从 API 读取和刷新。
- 错误状态有可理解的 UI 和 API 返回。

## 2. 当前基线观察

截至本文档创建时：

- 后端已经有 Fastify 模块边界和 Drizzle schema。
- `apps/api/src/db/client.ts` 会打开 SQLite，并设置 WAL 和 foreign keys。
- `apps/api/src/db/schema.ts` 已定义 `accounts`、`categories`、`transactions`、`transfers` 等表。
- `accounts`、`categories`、`transactions`、`reports` 路由当前主要返回 demo 固定数据。
- `transactions` 的 `POST/PUT/DELETE` 当前仍是 `moduleStatus(...)` 占位。
- 前端 `LedgerHome`、`EntryEditorPage` 主要依赖 `apps/web/src/lib/ledgerStore.ts` 的 localStorage 和静态数组。
- `apps/web/src/lib/api.ts` 目前只有 `apiGet`，尚无 `POST/PUT/DELETE` 封装。

这些不是测试失败，而是主线开发前的基线。主 Agent 打通 API 后，应按下方清单回归。

## 4. API 回归清单

### 4.1 健康检查

请求：

```powershell
Invoke-RestMethod http://localhost:3000/health
```

期望：

- HTTP 200。
- 返回 `data.status = ok`。

### 4.2 账户接口

请求：

```powershell
Invoke-RestMethod http://localhost:3000/api/accounts
```

期望：

- HTTP 200。
- 返回数组。
- 至少包含一个可记账账户。
- 字段包含：`id`、`name`、`type`、`kind`、`balance`、`color`、`icon`、`includeInAssets`。
- `balance` 应来自 SQLite 初始余额加流水合计，而不是硬编码 demo。

边界：

- 空库第一次启动时应有默认账户，或前端给出“请先创建账户”的空状态。
- 隐藏或归档账户不应出现在普通记账账户选择中。

### 4.3 分类接口

请求：

```powershell
Invoke-RestMethod http://localhost:3000/api/categories
```

期望：

- HTTP 200。
- 返回数组。
- 同时包含 `income` 和 `expense` 分类。
- 字段包含：`id`、`name`、`type`、`icon`、`color`、`sortOrder`。
- 前端记一笔分类网格应使用此接口结果。

边界：

- `type=income` 筛选只返回收入分类。
- `type=expense` 筛选只返回支出分类。
- 隐藏分类不应默认出现在记账网格。

### 4.4 新增支出

请求示例：

```powershell
$body = @{
  type = 'expense'
  happenedOn = '2026-05-31'
  amount = '12.34'
  accountId = '<account-id>'
  categoryId = '<expense-category-id>'
  memberId = $null
  note = '回归测试支出'
} | ConvertTo-Json

Invoke-RestMethod http://localhost:3000/api/transactions -Method Post -Body $body -ContentType 'application/json'
```

期望：

- HTTP 200 或 201。
- 返回新增账单 id。
- 数据库存储金额为负数：`-12.34`。
- `displayAmount` 为正数：`12.34`。
- `GET /api/transactions` 能查到该记录。
- 首页当月支出增加 `12.34`。
- 对应账户余额减少 `12.34`。
- 报表分类支出增加 `12.34`。

```powershell
  type = 'income'

  note = '回归测试收入'
```

期望：

- HTTP 200 或 201。
- 数据库存储金额为正数：`88.88`。
- `GET /api/transactions` 能查到该记录。
- 首页当月收入增加 `88.88`。
- 对应账户余额增加 `88.88`。
- 报表收入统计增加 `88.88`。

### 4.6 新增转账

请求示例：

```powershell
$body = @{
  happenedOn = '2026-05-31'
  amount = '50.00'
  fromAccountId = '<from-account-id>'
  toAccountId = '<to-account-id>'
  note = '回归测试转账'
} | ConvertTo-Json

Invoke-RestMethod http://localhost:3000/api/transfers -Method Post -Body $body -ContentType 'application/json'
```

期望：

- HTTP 200 或 201。
- 生成一条 `transfers` 主记录。
- 生成两条 `transactions`：
  - 转出账户：`type=transfer_out`，`amount=-50.00`。
  - 转入账户：`type=transfer_in`，`amount=50.00`。
- 两条流水共享同一个 `transferId`。
- 首页流水可显示为一笔转账，或显示有关联的转出/转入记录，但不能重复计入收支。
- 报表收入、支出不因转账变化。
- 转出账户余额减少 `50.00`。
- 转入账户余额增加 `50.00`。

边界：

- 转出账户和转入账户相同应返回 400。
- 金额为 0、负数、非数字应返回 400。

### 4.7 账单列表和合计

请求：

```powershell

Invoke-RestMethod 'http://localhost:3000/api/reports/summary?month=2026-05'
```

期望：

- `transactions` 按日期倒序分组。
- 每天小计收入/支出正确。
- 月收入只统计普通收入。
- 月支出只统计普通支出。
- 转账不进入收入/支出合计。
- 删除或软删除记录不进入任何合计。

### 5.5 错误状态

必须覆盖：

- 金额为空或 0：前端阻止提交，并显示提示。
- 金额格式错误：前端阻止提交，或后端返回 400 后前端显示错误。
- 未选择账户：不能提交。
- 未选择分类：收入/支出不能提交。
- 转账同账户：不能提交。
- 后端 500：前端不能白屏，应显示重试/错误文案。
- 网络断开：前端不能误报保存成功。
- API 返回字段缺失：前端不能崩溃，至少进入错误状态。

## 6. 持久化专项测试

### 6.1 刷新不丢

流程：

1. 新增一笔支出、一笔收入、一笔转账。
2. 复制三笔记录的备注和金额。
3. 浏览器刷新。
4. 再次进入首页、资产页、报表页。

期望：

- 三笔数据仍存在。
- 合计与刷新前一致。
- localStorage 清空后再次刷新，数据仍存在。

### 6.2 服务重启不丢

流程：

1. 新增测试数据。
2. 停止 API 服务。
3. 重新启动 API 服务。
4. 打开前端刷新。

期望：

- 数据仍存在。
- SQLite 文件未被重建覆盖。
- 账户余额和报表合计一致。

### 6.3 数据库文件位置

检查：

- 开发环境默认库应在 `./data/app.db` 或本轮指定的 `./data/test-persistence.db`。
- 群晖环境应支持 `DATABASE_URL=file:/data/app.db`。
- 数据库目录不存在时服务可自动创建目录。

### 9.1 P0 API 测试脚本


```powershell

```

另开 PowerShell 执行 API 验收：

```powershell
$health = Invoke-RestMethod "$base/health"
if ($health.data.status -ne 'ok') { throw 'health failed' }

$accounts = (Invoke-RestMethod "$base/api/accounts").data
if (-not $accounts -or $accounts.Count -lt 2) { throw 'accounts missing' }

$categories = (Invoke-RestMethod "$base/api/categories").data
$expenseCategory = $categories | Where-Object { $_.type -eq 'expense' -and $_.id -ne 'transfer' } | Select-Object -First 1
$incomeCategory = $categories | Where-Object { $_.type -eq 'income' } | Select-Object -First 1
if (-not $expenseCategory -or -not $incomeCategory) { throw 'categories missing' }

$accountA = $accounts[0]
$accountB = $accounts | Where-Object { $_.id -ne $accountA.id } | Select-Object -First 1

$expenseBody = @{
  type = 'expense'
  happenedOn = '2026-05-31'
  amount = 12.34
  accountId = $accountA.id
  categoryId = $expenseCategory.id
  note = 'P0 API 支出'
} | ConvertTo-Json
$expense = Invoke-RestMethod "$base/api/transactions" -Method Post -Body $expenseBody -ContentType 'application/json'
if (-not $expense.data.id) { throw 'expense create failed' }

$incomeBody = @{
  type = 'income'
  happenedOn = '2026-05-31'
  amount = 88.88
  accountId = $accountA.id
  categoryId = $incomeCategory.id
  note = 'P0 API 收入'
} | ConvertTo-Json
$income = Invoke-RestMethod "$base/api/transactions" -Method Post -Body $incomeBody -ContentType 'application/json'
if (-not $income.data.id) { throw 'income create failed' }

$transferBody = @{
  type = 'transfer'
  happenedOn = '2026-05-31'
  amount = 50
  fromAccountId = $accountA.id
  toAccountId = $accountB.id
  note = 'P0 API 转账'
} | ConvertTo-Json
$transfer = Invoke-RestMethod "$base/api/transactions" -Method Post -Body $transferBody -ContentType 'application/json'
if (-not $transfer.data.id) { throw 'transfer create failed' }

$transactions = (Invoke-RestMethod "$base/api/transactions").data
if (-not ($transactions | Where-Object { $_.note -eq 'P0 API 支出' })) { throw 'expense not listed' }
if (-not ($transactions | Where-Object { $_.note -eq 'P0 API 收入' })) { throw 'income not listed' }
if (-not ($transactions | Where-Object { $_.note -eq 'P0 API 转账' -and $_.type -eq 'transfer' })) { throw 'transfer not listed' }

$badTransferBody = @{
  type = 'transfer'
  happenedOn = '2026-05-31'
  amount = 1
  fromAccountId = $accountA.id
  toAccountId = $accountA.id
} | ConvertTo-Json
try {
  Invoke-RestMethod "$base/api/transactions" -Method Post -Body $badTransferBody -ContentType 'application/json'
  throw 'same-account transfer should fail'
} catch {
  if ($_.Exception.Response.StatusCode.value__ -ne 400) { throw 'same-account transfer did not return 400' }
}

$summary = (Invoke-RestMethod "$base/api/reports/summary").data
if (-not $summary.income -or -not $summary.expense) { throw 'summary missing' }

$accountsAfter = (Invoke-RestMethod "$base/api/accounts").data
if (-not $accountsAfter) { throw 'accounts after mutation missing' }
```

P0 通过标准：

- 所有请求成功或按预期返回 400。
- 新增收入、支出、转账刷新后仍能从 `GET /api/transactions` 查到。
- 账户余额变化与流水金额一致。
- `reports/summary` 不把转账计入收入/支出。

- 账户余额、月收支、转账统计与导入预览一致。

备份验收：

```powershell
Invoke-RestMethod http://localhost:3000/api/backups/create -Method Post
Get-ChildItem '.\data\backups'
Invoke-RestMethod http://localhost:3000/api/backups
```

期望：

- `data/backups` 中生成可识别的备份文件。
- `GET /api/backups` 能列出该备份。
- 备份文件至少包含 SQLite 数据库；后续如有上传附件，也要包含附件目录。

恢复验收：

```powershell
Invoke-RestMethod http://localhost:3000/api/backups/restore `
  -Method Post `
  -ContentType 'application/json' `
  -Body '{"fileName":"<backup-file>"}'
```

期望：

- 恢复前自动生成当前库备份。
- 恢复后服务可继续启动。
- 恢复后的账单、账户、报表数据与备份时一致。

## 10. 当前硬阻碍短清单

按“能不能部署给用户正常单人使用”的角度排序：

1. 导入仍是占位实现
   - `apps/api/src/modules/imports/routes.ts` 仍返回 scaffolded。
   - 不能解析口袋记账 `.xls`，用户历史数据无法迁移。
   - 这是从旧 App 过渡到本系统的 P0 阻碍。

2. 备份/恢复仍是占位实现
   - `apps/api/src/modules/backups/routes.ts` 仍返回 scaffolded。
   - 群晖长期使用必须能生成、列出、恢复备份。
   - 没有备份闭环前，不建议正式承载真实账本。

3. 登录保护仍是占位实现
   - `POST /api/auth/login` 直接返回成功。
   - `GET /api/auth/me` 返回 `authenticated: false`。
   - 没有 session 校验和路由保护。
   - NAS 局域网部署也会暴露个人财务数据，这是 P0 安全阻碍。

4. 编辑/删除账单仍是占位实现
   - `GET/PUT/DELETE /api/transactions/:id` 仍是 scaffolded。
   - 单人日常使用一定会录错账，没有编辑/删除会影响真实使用。

5. 分类、账户新增编辑仍是占位实现
   - 当前有默认种子账户和分类，但无法按用户真实账户体系维护。
   - 导入前可暂时不阻塞测试，正式迁移前会阻塞。

6. Docker/NAS 流程还缺一次真实验收
   - Dockerfile 和 compose 已具备基础形态。
   - 仍需要实际执行 `docker compose build/up`，确认 `better-sqlite3` 原生依赖、前端静态托管、`/data/app.db` 持久化都正常。
   - `docker-compose.yml` 依赖 `.env`，缺失时启动会失败，需要文档或脚本兜底。

7. 前端部分页面仍有 fallback/demo 数据风险
   - 资产页、报表页在 API 无数据时会回退到静态数据。
   - 部署验收时必须确认 API 失败不会展示假资产/假报表。

8. 数据库 bootstrap 与 Drizzle schema 没有统一迁移链
   - 当前有 `ensureDatabase()` 手写建表和种子数据。
   - `apps/api/drizzle` 只有 `.gitkeep`，没有真实迁移文件。
   - 短期可跑，长期升级需要明确迁移策略，否则 NAS 版本迭代容易出数据结构风险。
